# SPEC-0078: Receipt provenance footer

## Purpose

Make every full receipt identify its open-source home without turning a local terminal
receipt into an unsolicited network action. The terminal, SVG, and PNG share one quiet,
scheme-less repository line; PR Markdown and the self-contained PR artifact make the
same destination clickable. This serves I1 (no product-path fetch), I3 (attribution is
explicit), and I5 (the changed bytes are deliberately golden-gated). Design and
evidence: [`docs/internal/research/2026-07-10-receipt-branding-provenance.md`](../docs/internal/research/2026-07-10-receipt-branding-provenance.md).

## Invariants

I1: fixed text and user-activated links add no fetches or model calls. I2: no monetary
value is added or changed. I3: one canonical repository identity makes provenance
traceable. I4: telemetry and its payload stay unchanged. I5: every output byte change
is covered by focused assertions and deliberate goldens. I6: provenance makes no model
or agent ranking.

## Design

Keep the existing centered footer unchanged, then add one muted centered line directly
below it:

```text
aireceipts · local · npx aireceipts-cli
github.com/anandgupta42/receipts
```

The shared terminal/receipt AST uses the scheme-less display text. PR Markdown appends
an outer-fence attribution line: `Generated by [aireceipts on
GitHub](https://github.com/anandgupta42/receipts) · install: \`npx aireceipts-cli\``.
The artifact footer retains its existing provenance text and adds `<a
href="REPOSITORY_URL">source & install</a>`. Grocery keeps its thank-you stamp and
barcode, placing the two provenance lines below the barcode. This is attribution, not a
claim that a receipt is "powered by" another product.

## Requirements

- **R1 — Full receipt provenance.** Classic, grocery, and datavis `Block[]` views
  render the design's two lines in terminal, SVG, PNG, and `compare`. `week` renders
  the same two centered lines. Focused tests assert that Classic and Datavis place the
  centered muted display line directly after the footer, Grocery orders thank-you,
  barcode, install footer, display URL, every terminal line is at most 50 columns, and
  SVG contains display text but no `<a>`. The six-line `aireceipts templates` preview,
  mini receipt, handoff, statusline, JSON, and CSV outputs remain byte-identical.
- **R2 — One canonical identity.** `src/receipt/branding.ts` owns the repository URL,
  display URL, install footer copy, and the PR attribution copy as
  `REPOSITORY_URL`, `REPOSITORY_DISPLAY`, `INSTALL_FOOTER_TEXT`, and
  `PR_ATTRIBUTION_LINE`. Its test normalizes package metadata by removing `git+` and a
  trailing `.git`, then requires the resulting URL to equal `REPOSITORY_URL`. No receipt,
  PR, or artifact rendering source outside this module duplicates the repository URL literal.
- **R3 — PR Markdown attribution.** `aireceipts pr` adds the design's clickable
  attribution after the fenced receipt and after optional details, handoff, and artifact
  lines. Before `detailsSection` and `handoffSection` are budgeted,
  `renderPrBodyDetailed` reserves `[...PR_ATTRIBUTION_LINE].length + 1` in its existing
  `used` calculation; its last nonblank line is always that attribution. The fenced
  receipt uses R1's scheme-less footer.
- **R4 — Artifact attribution without remote loading.** Every PR artifact has exactly
  one user-activated canonical repository anchor in its footer (two with `--samosa`),
  with exact `href` allowlists `[REPOSITORY_URL]` and `[REPOSITORY_URL, SAMOSA_URL]`.
  It has no `<script`, `<link`, `<img`, `<iframe`, `@import`, `@font-face`, or CSS
  `url(`. This supersedes SPEC-0070's zero-link count: anchors require an explicit user
  action, unlike remote subresources. The footer retains existing artifact provenance.
- **R5 — Documentation and golden contract.** Update affected README and guides,
  `docs/guide/01-getting-started.md`, `04-read-a-receipt.md`, `05-compare.md`, and
  `06-week.md`; run `npm run docs:site`; and deliberately update all affected
  text/SVG/PNG goldens and focused assertions. Existing link-free SVG policy remains:
  the visible repository URL is text, not an SVG anchor.

**Kill criterion:** cut the second line if it cannot fit at the existing 50-column
receipt width in all three templates without wrapping, clipping, or special-case layout;
do not replace it with a shorter unverified brand phrase.

## Scenarios

- **Given** a priced Classic receipt, **when** rendered as terminal text, SVG, or PNG,
  **then** the unchanged install footer is followed by the centered display URL.
- **Given** a Grocery receipt, **when** rendered, **then** thank-you and barcode retain
  their order and the two provenance lines come after the barcode.
- **Given** `aireceipts pr` with details, handoff, and an artifact link, **when** its
  body is rendered, **then** the clickable attribution is the final nonblank line and
  the complete body is at most 65,000 characters.
- **Given** an artifact with `--samosa` off or on, **when** its HTML is inspected,
  **then** it has respectively one or two allowed user-activated anchors and no remote
  subresources.
- **Given** JSON, CSV, mini, handoff, or statusline output, **when** rendered before
  and after this feature, **then** its bytes are unchanged.

## Non-goals

- **"Powered by receipts" wording** — it is generic and describes the relationship
  inaccurately; the product name, repository, and installation command are clearer.
- **OSC 8 terminal hyperlinks or SVG anchors** — terminal capability detection and
  clickable exported SVG are outside this small provenance change; visible text works
  everywhere and preserves the SVG self-contained policy.
- **Telemetry, tracking parameters, a web fetch, or an install prompt** — I1 and the
  local-first product contract forbid them.
- **Branding mini, handoff, statusline, JSON, or CSV** — those are compact or
  machine-readable contracts, not full receipt surfaces.

## Validation

*2026-07-10 — S1 self, S2 independent Codex critic, S3 worth gate, S4 lint.*

- **S1:** I1–I6 are restated above. The design emits fixed local text and two
  user-activated links only; it computes no values, changes no telemetry, and makes no
  ranking. I5 is pinned by new focused ordering/width/link assertions plus goldens.
- **S2 (Codex) — accepted:** specified the I1–I6 restatement; exact PR-attribution
  reservation and no-details/boundary coverage; canonical package URL normalization;
  explicit artifact `href` allowlists and subresource bans; template-preview exclusion;
  actual affected docs; and 50-column/SVG ordering checks. **Rejected:** no expanded
  surface or alternative wording; they add capability or ambiguity without advancing
  detached-receipt provenance.
- **S3 — worth:** users who share receipts in terminal captures, SVG/PNG exports, PR
  comments, and artifacts encounter detached provenance every shared session; this is
  recurring, not a cleanup-only event. Do nothing leaves a copyable cost receipt with no
  direct project or install route. A README-only change helps direct visitors but not
  detached receipts; terminal/SVG-only branding misses the requested clickable PR and
  artifact route. Strongest cut case: the existing install footer already names the
  package, so another 32-character line may be visual noise. The kill dry-run survives:
  `github.com/anandgupta42/receipts` is 32 code points, below the 50-column contract;
  implementation must still prove every template. **Verdict: build now.**
- **S4:** `node scripts/spec-lint.mjs specs/SPEC-0078-receipt-provenance.md` passed.
