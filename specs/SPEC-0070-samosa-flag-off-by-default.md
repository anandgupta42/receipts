# SPEC-0070: Samosa link opt-in

Invariants: I3 (the footer stays an honest, disclosed line whether or not the tip link is
shown — nothing about cost or methodology changes), I5 (goldens gate the artifact bytes; the
default artifact footer becomes link-free and its golden is regenerated). I2/I6 are untouched —
the samosa link is a tip affordance, never a dollar figure or a ranking.

## Purpose

SPEC-0034 puts a `buy me a samosa` tip link on every PR-posted surface — the comment's
`<details>` section (`body.ts:523,563`) and the artifact page footer (`html.ts:113`). The
maintainer wants that link **off by default** on the surfaces aireceipts posts onto a PR, behind
an opt-in flag, so a posted receipt carries no tip link unless explicitly enabled. This is a
default-behavior change, not a removal: the flag restores the exact prior output, and the
standalone tip page (`site/samosa.html`) and the marketing docs-site footer are out of scope
(they are the project's own pages, not something posted onto another repo's PR).
**Kill criterion:** if gating the artifact link weakened its self-contained-page property
(SPEC-0034 R3's "exactly one external reference"), this would be reconsidered — but off → *zero*
external references, a property `artifact.test.ts` asserts directly by counting `https://`
occurrences, so the criterion is satisfied and locally verifiable.

## Requirements

- **R1 — opt-in `--samosa` flag, default off.** `parseOptions` gains a boolean `samosa`
  (`CliOptions.samosa`), false unless `--samosa` is passed, threaded into `PrOptions.samosa`.
  It parallels the existing opt-in booleans (`--artifact`, `--share`): a bare presence flag that
  turns a surface ON, so the default (absent) is off.
- **R2 — the comment tip link is gated.** `detailsSection` renders the closing `SAMOSA_LINK`
  (and reserves its bytes in the size-cap frame) **only** when samosa is enabled; off, the
  `<details>` section closes straight after the last receipt with no link and no reserved bytes.
  A comment with no details section already has no link (unchanged).
- **R3 — the artifact footer is gated.** With samosa off (default), the artifact footer is the
  plain line `aireceipts · local · generated by aireceipts pr --post --artifact` — no `<br>`,
  no glyph, no `<a>`. With samosa on, it is the SPEC-0034 footer (glyph + `buy me a samosa`
  link). Off makes the artifact page have **zero** external references (strengthening R3's
  self-contained property); on preserves the single intentional link.
- **R4 — the flag round-trips through the ref payload; no schema-version bump.** `PrBodyExtras`
  gains an optional `samosa?: boolean`, and the strict `prBodyExtrasSchema` (`sanitize.ts:212`) is
  extended with `samosa: z.boolean().optional()` so a payload carrying it is accepted. The producer
  **omits the field when off** — it writes `samosa: undefined` (like `artifactLink`/`handoff`), never
  `false` — so a default `store=ref` payload carries **no new key** and stays byte-identical to a
  pre-feature ref (an older strict consumer still accepts it; only a `--samosa` ref grows the
  field). A `--samosa` ref persists `samosa: true`, so `postRef`'s CI re-render reproduces the tip
  link. Because the default payload is unchanged and the field is additive-optional, **no
  `PR_RECEIPT_SCHEMA_VERSION` bump** is taken (the `payloadTypes.ts` policy comment is refined to
  say additive-optional needs none). An existing pre-feature ref (no field) still deserializes and
  re-renders under the **current default (off)** — the intended new behavior, not a frozen replay of
  the old link (a renderer default change is not a round-trip violation).
- **R5 — goldens + tests.** The artifact golden (`goldens/html/pr-artifact.html`) is regenerated
  to the default (link-free) footer. Tests cover: `--samosa` parsing **and its end-to-end thread
  through `pr` → `PrOptions` → both renderers**; the comment link present only with the flag and
  absent by default; the artifact footer plain by default (no `<br>`, no glyph, no `<a>`, zero
  `https://` refs) and the single link with the flag; and the ref round-trip preserving `samosa`
  (and an omitted-field payload deserializing as off).

## Scenarios

- **Given** `aireceipts pr --post` (no `--samosa`), **when** the comment renders with a details
  section, **then** the section closes with no `buy me a samosa` link.
- **Given** `aireceipts pr --post --artifact`, **when** the artifact renders, **then** the footer
  is the plain install line and the page has zero external (`https://`) references.
- **Given** `aireceipts pr --post --samosa`, **when** the comment renders, **then** the details
  section closes with the `buy me a samosa` link (SPEC-0034 shape restored).
- **Given** `aireceipts pr --post --artifact --samosa`, **when** the artifact renders, **then**
  the footer carries the glyph and the single samosa link.
- **Given** a `store=ref` payload written with `--samosa`, **when** `postRef` re-renders it,
  **then** the comment reproduces byte-for-byte (link present); a payload without the field
  deserializes as off.

## Non-goals

- **Removing the samosa feature or the tip page.** `site/samosa.html` and the glyph stay; this
  is a default flip, not a deletion. `--samosa` restores the exact prior output. The tip page,
  the drawn glyph, and its single-source guard (`svg.test.ts`) remain **SPEC-0034's** to own —
  this spec does not re-test non-PR-posted surfaces.
- **Gating the marketing docs-site footer** — that is the project's own page, not a PR-posted
  surface, and is out of scope.
- **A config-file / env-var control.** v1 is a single CLI flag on `pr`, consistent with the other
  opt-in surface flags; a persisted preference is a separate concern if it ever proves needed.
