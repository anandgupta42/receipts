# Receipt branding and repository provenance

Date: 2026-07-10

## Question

What attribution should appear at the bottom of an aireceipts receipt so a
person who encounters a pasted, posted, or exported receipt can identify the
project and reach its install path without turning the receipt into an ad?

## Recommendation

Keep the shipped first line exactly:

```text
aireceipts · local · npx aireceipts-cli
```

Add one quiet, centered line directly below it:

```text
github.com/anandgupta42/receipts
```

Do not use “Powered by receipts.” `receipts` is a generic noun and the
repository slug; the product, command, and package are respectively
`aireceipts`, `aireceipts`, and `aireceipts-cli`. “Powered by” also
implies that the tool is an underlying service. The existing line already
states truthful provenance, local-first posture, and the install command; the
new line supplies the missing discovery path.

The two lines are 39 and 32 Unicode code points respectively, so each fits the
50-column receipt contract. The repository line is deliberately scheme-less in
shared receipt blocks. It remains readable and copyable in terminals, code
fences, SVGs, and PNGs, and it avoids introducing `https://` bytes into the
artifact's embedded `<pre>` receipts.

Where a surface natively supports links, add one explicit, descriptive link
outside the receipt data:

```markdown
Generated by [aireceipts on GitHub](https://github.com/anandgupta42/receipts) · install: `npx aireceipts-cli`
```

That line belongs once at the end of a posted PR comment, outside every fenced
receipt. The artifact HTML footer should instead use a compact
`source & install` anchor. Terminal output, SVG, and PNG remain visible text
only; do not add OSC 8 control sequences or SVG anchors in this change.

## Why this is the right shape

### It is provenance, not promotion

The closest generated-report precedents use factual attribution in a quiet
footer:

- [Criterion.rs](https://github.com/criterion-rs/criterion.rs/blob/60ab5fd10cc41d5c43a421f982d6bd981d36d05f/src/html/benchmark_report.html.tt#L301-L304)
  ends its benchmark report by saying it was generated by Criterion.rs and
  links the name to its repository.
- [Istanbul](https://github.com/istanbuljs/istanbuljs/blob/28ffdbc314596bdcb3007e85d30a62372602b262/packages/istanbul-reports/lib/html/index.js#L97-L105)
  styles its generated-by footer as quiet, centered, and small.
- [Lighthouse](https://github.com/GoogleChrome/lighthouse/blob/3608d82ba7ee4912e5ba9af9eeb178168ec08b62/report/assets/templates.html#L455-L475)
  centers a compact generated-by/version footer and gives one practical
  follow-up link.
- [Codecov](https://github.com/codecov/umbrella/blob/d76746309ae4e5f75381588b76bf4292865c9520/apps/worker/services/notification/notifiers/mixins/message/sections.py#L246-L267)
  separates a “Powered by” footer from report data; its newer version makes
  the next useful action the dominant footer link.

These examples support one small provenance area, not a second promotional
section. The recommendation preserves the product's existing footer and adds
only the information a reader cannot otherwise recover.

### The repository is the real install funnel

The public [repository](https://github.com/anandgupta42/receipts) identifies
aireceipts as a local deterministic receipt tool and puts
`npx aireceipts-cli` and global installation in the first screen of its
README. A repository address therefore both identifies the source and leads
to the maintained install instructions. No tracking URL or redirect is needed.

### Link behavior must follow the surface

The current PR body places its receipts in fenced code blocks. GitHub documents
that normal Markdown URLs are autolinked, but that behavior cannot turn a
literal code-fence line into an actionable product footer. Put the one
Markdown link outside the fence instead. It will be clickable once per posted
comment, rather than repeated below the rollup and every expanded receipt.

The standard permits links in SVG, but they are activated only by user action.
That is not enough reason to add them here: PNG cannot carry the interaction,
some image hosts sanitize or flatten SVG, and the project deliberately keeps
receipt SVGs self-contained. A visible URL works in both raster and vector
exports. OSC 8 is also not appropriate: iTerm2 describes it as a non-standard
extension and warns that proprietary escape sequences can misbehave through
tmux or screen. Conditional terminal link bytes would also complicate I5's
byte-stable contract.

For HTML and Markdown, use descriptive link text, not the raw URL as the
anchor. W3C's link-purpose guidance says a URI alone is generally not
sufficiently descriptive for users navigating a list of links.

## Existing product decision and gap

This is an amendment, not a net-new footer:

- SPEC-0055 intentionally changed the default receipt-card footer to
  `aireceipts · local · npx aireceipts-cli`, specifically because receipts
  circulate on public PRs. It also explicitly kept terminal receipts
  link-free.
- The live PR #189 receipt demonstrates the limitation: its footer is useful
  but sits inside a code fence, so it provides no repository destination.
- Classic and datavis use `FOOTER_TEXT` in `src/receipt/present.ts`; the
  PR rollup has a duplicate `FOOTER_TEXT` in `src/pr/body.ts`; `week`
  hard-codes the same line separately. Grocery ends in a thank-you line and
  barcode, so it currently lacks both the install CTA and repository address.
- Current golden coverage is broad: 60 of 104 golden artifacts contain the
  current install footer, while 14 grocery goldens carry the thank-you footer.

The proposed repository line closes the discovery gap without discarding the
already-reviewed local and install claims.

## Surface policy

| Surface | Treatment | Why |
|---|---|---|
| Default/detail/template terminal receipt | Existing install line + scheme-less repo line | Copyable and screenshot-safe; no terminal capability detection. |
| SVG and PNG | Same two visible lines | One block model preserves text/SVG parity; PNG inherits SVG. |
| Compare | Inherited on each receipt card | Each side can be copied or shared independently. |
| Week | Same two lines in its separate renderer | It is a shareable digest with its own footer. |
| Backfill files and demo | Inherited full receipt branding | They render the canonical receipt. |
| PR fenced rollup and expanded receipts | Same two visible lines | Preserves receipt parity and provenance when copied elsewhere. |
| PR comment outer body | One explicit Markdown attribution link after all optional sections | This is the clickable discovery affordance; it is not repeated per receipt. |
| PR artifact HTML chrome | One explicit `source & install` anchor | The artifact is a browser-native, shareable surface. |
| JSON and CSV | No change | Strict machine contracts have no visual bottom; provenance metadata would be a separate schema decision. |
| Mini receipt, handoff, statusline, setup, integrations | No change | Mini has a fixed six-line ambient contract; handoff is not a second receipt; the rest are not receipt cards. |

## Implementation proposal

This deserves a new Draft spec because it changes golden-gated output and
supersedes parts of shipped SPEC-0055 and SPEC-0070.

1. Add `src/receipt/branding.ts` as the one code-level source of truth for:
   - `REPOSITORY_URL = "https://github.com/anandgupta42/receipts"`
   - `REPOSITORY_DISPLAY = "github.com/anandgupta42/receipts"`
   - the existing install footer line
   - Markdown and HTML attribution copy where appropriate.

   Add a parity test against the normalized `package.json.repository.url` so
   a future repository move cannot silently leave receipt output stale.

2. Keep the policy in the existing `Block[]` presentation model; do not
   append content inside `renderBlockLines()` and do not place branding in
   `ReceiptModel`. For classic, datavis, and PR blocks, emit the existing
   `footer` block followed by a centered muted `note` holding
   `REPOSITORY_DISPLAY`. This needs no new block kind and automatically
   reaches terminal, SVG, and PNG renderers.

3. Make grocery end with the same two provenance lines after its thank-you and
   barcode, retaining its visual theme while ensuring the actual bottom of
   every template identifies the project. The implementation must keep one
   deterministic ordering for terminal and SVG rendering.

4. Replace the separate constants in `present.ts`, `body.ts`, and
   `week.ts` with the shared values. The affected paths are:
   - `src/receipt/present.ts` (`tailBlocks`, classic, grocery, datavis)
   - `src/pr/body.ts` (`prBlocks`, `renderPrBodyDetailed`)
   - `src/receipt/week.ts`
   - `src/pr/html.ts`.

5. Add the Markdown attribution after the fenced receipt, optional details,
   handoff, and artifact link in `renderPrBodyDetailed()`. Include its exact
   fixed byte length in that function's existing 65k comment-budget
   calculation.

6. Change the default artifact footer from an unlinked statement to one
   allowlisted `<a href="https://github.com/anandgupta42/receipts">`
   anchor. Amend SPEC-0070's “zero external references” test: it currently
   counts every `https://` byte, including harmless literal text in
   `<pre>`. The replacement test should permit exactly the one canonical,
   user-activated repository anchor by default and, with `--samosa`, exactly
   that anchor plus the existing samosa anchor. It must still forbid scripts,
   stylesheets, images, iframes, and unallowlisted external URLs.

7. Do not add OSC 8, URL shorteners, QR codes, telemetry, network calls,
   configuration switches, JSON/CSV fields, or an opt-out flag in v1.

## Required verification and acceptance criteria

The new spec should restate I1–I6 and require:

- Every classic, datavis, grocery, PR, week, SVG, PNG, compare, demo, and
  backfill receipt displays the two-line provenance treatment in a stable
  order. Mini, handoff, statusline, JSON, and CSV remain byte-identical.
- The terminal and SVG block lists stay in parity; neither has OSC 8 or an
  SVG `<a href>`.
- A PR comment has exactly one explicit repository Markdown link outside code
  fences, even when it contains a rollup, details, a handoff, and an artifact
  link.
- The default artifact contains exactly one allowlisted repository anchor and
  no external subresources. `--samosa` adds only its existing, deliberate
  destination.
- `README.md`, guides 01/04/05/06/10, and generated `site/docs/*.html`
  show the new receipt bytes. The README guard stays green.
- Goldens change once, intentionally, with a committed explanation.

Run the repository's complete verification block without piping any command:


## Rejected alternatives

| Alternative | Reason not selected |
|---|---|
| `Powered by receipts · github.com/anandgupta42/receipts` | Generic product name, imprecise relationship, and 54 columns. |
| Replace the current footer with `aireceipts · github.com/anandgupta42/receipts` | A clean 45-column fallback, but it discards the deliberate local and direct-install claims from SPEC-0055. |
| Full `https://` URL in receipt blocks | Repeats in PR artifact `<pre>` content and breaks the current coarse external-reference test; the scheme-less display is sufficient there. |
| OSC 8 terminal hyperlinks | Non-standard capability, fragile through multiplexers, invisible after copy/paste, and adds conditional control bytes. |
| SVG hyperlinks | Technically valid but inconsistent with PNG and the current self-contained SVG contract; a visible URL is more portable. |
| JSON/CSV generator field | Requires a deliberate versioned data-contract decision; no visual footer exists there. |
| QR code or a second promotional panel | Adds visual weight without improving a copyable, already-short repository address. |
